# 株式会社a.s.ist｜ソフトウェア開発事例サイト（GitHub Pages）要件定義書

- 文書種別: 要件定義書（ブラッシュアップ版）
- 対象: 「ソフトウェア開発事例 / ソリューション紹介」Webサイト
- 公開方式: GitHub Pages（静的サイト）
- 最終更新日: 2026-01-31（JST）

---

## 1. 背景・目的

### 1.1 背景
a.s.ist のソフトウェア開発ソリューションを、顧客課題と提供価値が短時間で理解できる形で提示し、問い合わせ獲得につなげる。

### 1.2 目的
- **認知・理解**: 「どの業界のどんな課題に効くか」を明確にする  
- **信頼形成**: 事例・機能・アウトプット例（画像）を通じ、導入後イメージを提示する  
- **CV（問い合わせ）増**: 各ページに一貫したCTA（問い合わせ導線）を設置する  

### 1.3 前提
- GitHub Pages は静的ホスティングであり、サーバーサイド処理（DB・API・メール送信など）を標準では提供しない  
- お問い合わせフォームは外部サービス連携（例: Formspree 等）を採用する  

---

## 2. スコープ

### 2.1 In Scope（対象）
以下 4 つのソリューション詳細ページ（既存ページがある前提で、**統一テンプレート**へ揃える）

1) スペクトル自動解析ソフトウェア  
2) 異常検知ソフトウェア  
3) 報告書・レポート自動作成ソフトウェア  
4) データ解析アルゴリズム開発  

各ソリューションページには、以下の説明項目を必須とする（**固定の見出し**で統一）。
- 顧客の業界
- 解決する課題
- ソフトウェアの機能説明
- お問い合わせ（CTA）

加えて以下を含む。
- 画像（数点）掲載
- サイト共通 UI（#0136D8 基調、モダンでシンプル、分かりやすい）

### 2.2 Out of Scope（初期リリース対象外）
- 会員機能、ログイン、課金、SaaS提供
- 自前バックエンド（サーバー）運用
- 個人情報や機密情報の“直接”授受（フォーム入力項目は最小化し、外部フォームに委譲）

---

## 3. 想定ユーザー・利用シナリオ

### 3.1 想定ユーザー
- 製造業 / 材料 / 化学 / 分析計測 / 研究開発の現場  
  - R&D責任者、解析担当、DX推進、品質保証、現場責任者 など

### 3.2 主要シナリオ（ユーザーストーリー）
- **U1**: トップページで「自社課題に近いソリューション」を素早く発見する  
- **U2**: ソリューション詳細で「業界→課題→機能→アウトプット」を理解し問い合わせる  
- **U3**: 画像で導入後の成果物・UIを把握し、相談内容を具体化する

## 4. 情報設計（サイトマップ / URL設計）

### 4.1 ページ一覧
- `/` トップ（価値訴求・ソリューション導線・CTA）
- `/solutions/` ソリューション一覧（4件のカード）
- `/solutions/spectrum/` スペクトル自動解析
- `/solutions/anomaly-detection/` 異常検知
- `/solutions/report-automation/` 報告書・レポート自動作成
- `/solutions/algorithm-development/` データ解析アルゴリズム開発
- `/contact/` お問い合わせ
- `/privacy/` プライバシーポリシー（外部フォーム連携時は必須に近い）

### 4.2 ナビゲーション
- ヘッダー: `Solutions` / `Contact`
- 主要CTA: `お問い合わせ` ボタン（常時表示・ヘッダー右）
- フッター: `Privacy` / 会社情報 / 著作権

## 5. 画面要件（UI/UX）

### 5.1 デザインコンセプト
- 基調色: **#0136D8（Primary）**
- キーワード: **モダン / 余白 / 情報階層が明快 / 過度な装飾なし**
- 目標: “初見でも迷わず” ソリューション理解→問い合わせ

### 5.2 カラールール
- Primary: #0136D8  
- Neutral: 白～グレー（背景/区切り/本文）  
- 状態色: Success / Warning / Error は最小限に使用（フォーム/アラート用）
- **アクセシビリティ**: テキストと背景のコントラストは **4.5:1 以上**（原則）  
  - 大きな文字は 3:1 でも要件を満たすが、基本は4.5:1を狙う  

### 5.3 タイポグラフィ
- 見出し: 明確な階層（H1/H2/H3）  
- 本文: 読みやすい行間・文字サイズ（スマホで詰まらない）  
- 英数字と日本語の混在に配慮（数字は可読性優先）

### 5.4 レスポンシブ要件
- 320px〜（スマホ） / 768px〜（タブレット） / 1024px〜（デスクトップ）
- ソリューションカードは、画面幅に応じて 1→2→4 列に変化

### 5.5 共通レイアウト（全ページ）
- ヘッダー: ロゴ、ナビ、CTA
- フッター: 会社情報、Privacy、著作権
- Solutions配下はパンくず表示

## 6. ページ要件

### 6.1 トップページ（`/`）
**必須要素**
- ヒーロー（価値訴求 1〜2行 + CTA）
- ソリューションカード（4件）
  - タイトル / 1〜2行の説明 / 代表画像（任意） / 詳細リンク
- 事例の見せ方（最小）
  - 「業界 × 課題」タグで 2〜6件程度（将来拡張を考慮した構造）
- CTAセクション（ページ下部）
  - 「まずは課題をご相談ください」＋問い合わせボタン

### 6.2 ソリューション一覧（`/solutions/`）
- 4ソリューションをカードで一覧表示
- 絞り込みは初期リリースでは不要（将来拡張可能な設計に留める）

### 6.3 ソリューション詳細（`/solutions/*`）
**全ソリューションで共通テンプレ**にする（比較しやすさ・運用容易性を優先）。

#### セクション構成（必須）
1. **概要**（2〜4行）＋代表画像  
2. **顧客の業界**（箇条書き）  
3. **解決する課題**（Before/After で 2〜5点）  
4. **ソフトウェアの機能説明**  
   - 機能カード or アコーディオン
   - 入出力仕様（入力データ形式、出力レポート形式、連携先）を明記  
5. **お問い合わせ（CTA）**  
   - 相談の例（箇条書き）＋ボタン/フォーム  

#### 推奨（任意・可能なら入れる）
- 導入効果（定量/定性）※公開可能範囲で  
- 技術要素（抽象度を上げた説明: 例「統計的異常検知」「時系列解析」「自動レポーティング」など）

### 6.4 お問い合わせ（`/contact/`）
- 外部フォーム連携（例: Formspree）
- 入力項目は最小化（例）
  - お名前（任意）
  - 会社名
  - メール
  - 相談内容（必須）
- 送信後サンクス表示（別ページ or インライン）

### 6.5 プライバシー（`/privacy/`）
- 収集する情報の範囲、利用目的、外部フォームサービス利用の明示
- 問い合わせ窓口（メール等）

## 7. 機能要件（Functional Requirements）

### 7.1 コンテンツ運用
- **F-01**: ソリューションページをテンプレートで量産可能（Markdown/JSON 等データ駆動）
- **F-02**: 画像を `assets/` 配下で管理し、ページから参照
- **F-03**: OGP / meta description / title をページ単位で設定可能
- **F-04**: 404 ページの用意（任意だが推奨）

### 7.2 お問い合わせ
- **F-10**: フォーム送信が可能（外部サービス）
- **F-11**: 送信完了表示（サンクス）

---

## 8. 非機能要件（Non-Functional Requirements）

### 8.1 パフォーマンス
- 画像は WebP 等で圧縮・最適化し、可能なら遅延読み込みを採用
- 重要画像（ヒーロー等）は表示速度を優先し適切なサイズで配信

### 8.2 アクセシビリティ
- コントラスト比: 4.5:1（原則）
- 画像 alt を必須化
- キーボードフォーカス可視化

### 8.3 セキュリティ / コンプライアンス
- 公開リポジトリで機密情報を含めない
- フォームで機密情報（パスワード/カード番号等）を扱わない
- 外部フォーム連携時は Privacy で説明する

### 8.4 GitHub Pages 制約（運用上の制限）
- リポジトリ推奨上限: 1GB  
- 公開サイト上限: 1GB  
- デプロイ: 10分でタイムアウト  
- 帯域: 月 100GB（ソフト制限）  
- ビルド: 1時間 10回（ソフト制限、ただしカスタム Actions では扱いが変わる）

## 9. 技術要件（推奨アーキテクチャ）

### 9.1 方式（いずれかを採用）
- **A案: Jekyll（Markdown運用）**  
  - GitHub Pages との相性が良く運用が簡単  
- **B案: Astro / Next.js Static Export 等（コンポーネント運用）**  
  - UI統一・再利用に強い、ただし構成がやや増える  

※初期はA案（Jekyll）を推奨。GitHub Docs でも GitHub Actions によるデプロイが推奨されている。

### 9.2 ディレクトリ構成（例）
```
/
  index.html (or index.md)
  solutions/
    index.md
    spectrum.md
    anomaly-detection.md
    report-automation.md
    algorithm-development.md
  contact/
    index.md
  privacy/
    index.md
  assets/
    images/
      ...
  _includes/ (テンプレ部品)
  _layouts/  (ページテンプレ)
  _data/     (任意: JSON/YAML でカード情報など)
```

### 9.3 CI/CD
- GitHub Actions でビルド→GitHub Pages へデプロイ

### 9.4 カスタムドメイン（任意）
- Apex（example.com）と www（www.example.com）は両方設定可能  
- Apex 使用時、www サブドメイン併用が推奨されるケースがある（運用方針として明記）

## 10. コンテンツ仕様（ソリューションページの記入テンプレ）

各ソリューションページは以下を埋めて作成する。

### 10.1 概要
- 2〜4行で「誰のどの課題をどう解くか」
- 代表画像（UI/アウトプット例）

### 10.2 顧客の業界
- 箇条書き（例: 材料、化学、電池、半導体、分析受託、研究機関…）

### 10.3 解決する課題
- Before: 現状の課題（例: 手作業、属人化、再現性不足、解析に時間）
- After: 解決後（例: 自動化、標準化、再現性、短納期）

### 10.4 ソフトウェアの機能説明
- 機能（例: データ取り込み→前処理→解析→可視化→レポート）
- 入力形式（例: CSV、装置出力フォーマット、画像、時系列）
- 出力形式（例: PDF、PPTX、HTML、CSV）
- 連携（任意: LIMS、クラウドストレージ等）

### 10.5 お問い合わせ
- 相談例（箇条書き）
- CTA ボタン（Contact へ遷移、またはフォーム）

---

## 11. 受け入れ基準（Acceptance Criteria）

- **A-01**: 4つのソリューション詳細ページが共通テンプレで統一され、指定4項目とCTAが存在する  
- **A-02**: UI基調色 #0136D8 を中心に、モダンでシンプルなデザインになっている  
- **A-03**: 画像が表示され、alt が設定されている  
- **A-04**: Contact から問い合わせ送信でき、サンクス表示まで到達できる（外部フォーム）  
- **A-05**: GitHub Pages の制限（サイズ/ビルド等）を超えない  

---

## 12. 未確定事項（保留）

- 公開可能な導入効果の数値（KPI）有無
- 日本語のみ / 英語併記（多言語化）の要否
- お問い合わせ手段の最終決定（Formspree / Google Apps Script / mailto 併用）
- アクセス解析（GA4 等）導入の有無（Privacy 記載含む）
